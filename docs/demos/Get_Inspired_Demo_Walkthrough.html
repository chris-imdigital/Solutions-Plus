Chapter 1

How the feature actually took shape

This wasn't "write a prompt, get a feature." The build had real discovery behind it — evaluating approaches, ruling out dead ends, and documenting the decisions that mattered before any code was written.

Step 1
A vague concept surfaces
Client describes a "Design Shop" idea in a creative call — curated lifestyle photos paired with products. Pinterest-meets-shop-the-look. No technical path, no data model, no implementation plan. Just a direction.
Step 2
Evaluate data model options
Blogs? Collections? Custom pages? Metaobjects? Each was considered for how it handles product references, URL routing, admin editing, and SEO. Metaobjects won — clean semantic model, native product references, and standalone URL routing without a custom app.
Step 3
Solve the filtering problem
The Storefront API doesn't support field-level filtering on metaobjects. Server-side filtering would require a custom app or proxy. For a curated gallery under 250 entries, client-side filtering is fast, simple, and avoids infrastructure. Decision: Liquid renders everything, JS handles show/hide.
Step 4
Build a working POC on a dev store
Architecture validated hands-on: metaobject definition created via GraphQL, test entries populated, Liquid templates wired up, client-side JS written. Two platform gotchas discovered — self-referencing fields need a two-step mutation, and onlineStore capability isn't supported in TOML config.
Step 5
Write the feature spec
Everything distilled into one document: data model, field definitions, capabilities, category choices, theme architecture, frontend behavior, gotchas, POC-vs-production gaps, and a 9-step implementation sequence. This is the artifact Claude Code builds from.
Step 6
Design the visual direction in Figma
Design team creates Figma frames for the index page (masonry grid, filter tabs, product swatches, load more) and detail page (sticky hero, product grid, related entries). These become the visual source of truth for the frontend build.
Step 7
Hand off to Claude Code with MCPs
One prompt kicks off the implementation: "Let's use the Figma MCP, Shopify Dev MCP and the get inspired spec doc to start building out this feature." Claude reads the spec, pulls Figma designs, explores the codebase, plans the architecture, and builds 9 files across two phases.
Chapter 2

Two phases, two conversations with Shopify

With the spec and Figma designs in hand, Claude Code splits the implementation into two distinct phases. Each talks to a different part of Shopify through different MCP tools — but both are driven by the same spec.

🛠️
Phase 1

Data Layer

Create the metaobject definition, configure capabilities, upload images, populate test entries. Purely store configuration — no theme code.

Shopify Dev MCP GraphQL Admin API
1 Create metaobject definition (6 fields, 3 capabilities)
2 Add self-referencing field (two-step gotcha)
3 Upload hero images via staged uploads
4 Create test entries across categories
5 Wire product refs + related entries
6 Verify in Shopify admin
🎨
Phase 2

Frontend

Build Liquid sections, JS module, and templates that render the data. Figma provides pixel-level specs. The frontend reads from what Phase 1 created.

Figma MCP Shopify Dev MCP Theme Codebase
1 Read spec + pull Figma design context
2 Explore existing theme patterns
3 Load Shopify Liquid API context
4 Plan architecture + file inventory
5 Build sections, snippets, JS, templates
6 Compile JS via esbuild
7 Validate theme code

Chapter 4

Three patterns worth showing

Out of 9 files and ~1,000 lines, these three patterns best illustrate how the data layer and frontend come together.

Pattern
Masonry distribution algorithm
Cards distributed round-robin across 4 flex columns. Each column cycles through a different aspect-ratio pattern for visual variety.
inspiration-grid.js:10-16
var SIZE_PATTERNS = [
  ['tall', 'medium', 'short', 'xtall'],   // Col 1
  ['medium', 'xtall', 'tall', 'short'],    // Col 2
  ['xtall', 'short', 'medium', 'tall'],    // Col 3
  ['short', 'medium', 'tall', 'xtall']     // Col 4
];
Handoff
Liquid renders, JS distributes
All cards server-rendered into a hidden pool for SEO. JS clones them into visible columns — zero API calls after page load.
inspiration-grid.liquid:198-206
<div data-card-pool>
  {%- paginate shop.metaobjects.inspiration.values by 250 -%}
    {%- for entry in shop.metaobjects.inspiration.values -%}
      <div data-entry-id="...">
        {% render 'inspiration-card', entry: entry %}
      </div>
    {%- endfor -%}
  {%- endpaginate -%}
</div>
Layout
Sticky hero image
Hero image stays pinned while the user scrolls through the product grid. Top offset accounts for the theme's sticky header.
inspiration-hero.liquid:28-36
@media screen and (min-width: 750px) {
  .inspiration-hero__image-wrapper {
    position: sticky;
    top: calc(var(--header-height, 60px) + 20px);
    width: 562px;
    height: 734px;
    align-self: flex-start;
  }
}

Chapter 6

What this demonstrates

01
Full-stack with one tool

02
Spec-driven development
Four weeks of discovery distilled into one spec. Claude builds from the spec, not from vibes.
